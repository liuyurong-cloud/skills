# AI驱动研发工作流

自动化执行软件研发的完整生命周期，根据需求复杂度智能选择执行路径，确保高质量交付。

## 工作流程概览

```
                    ┌─────────────────┐
                    │   需求输入      │
                    └────────┬────────┘
                             ▼
                    ┌─────────────────┐
                    │  复杂度判定     │
                    │  (自动评估)     │
                    └────────┬────────┘
                             │
              ┌──────────────┴──────────────┐
              │                             │
              ▼                             ▼
    ┌─────────────────┐           ┌─────────────────┐
    │   简单需求      │           │   复杂需求      │
    │  (快速通道)     │           │  (完整流程)     │
    └────────┬────────┘           └────────┬────────┘
             │                             │
             ▼                             ▼
    ┌─────────────────┐           ┌─────────────────┐
    │  1. 快速开发    │           │  1. 需求分析    │
    │  2. 基础测试    │           │  2. 技术设计    │
    │  3. 直接交付    │           │     ⬇ 人工审查  │
    └─────────────────┘           │  3. 代码开发    │
                                  │  4. 全面测试    │
                                  │  5. 规范交付    │
                                  └─────────────────┘
```

## 复杂度判定

### 判定维度

评估以下维度，累计得分判定复杂度：

| 维度 | 简单 (1分) | 中等 (2分) | 复杂 (3分) |
|------|-----------|-----------|-----------|
| **代码量** | < 50行 | 50-200行 | > 200行 |
| **模块数** | 单一模块 | 2-3个模块 | > 3个模块 |
| **依赖关系** | 无外部依赖 | 1-2个依赖 | 多个依赖/第三方API |
| **业务逻辑** | 纯CRUD/简单计算 | 条件分支/状态机 | 复杂算法/多系统交互 |
| **数据模型** | 单一实体 | 2-3个关联实体 | 复杂关系/分布式事务 |
| **技术栈** | 熟悉的技术 | 1个新技术 | 多个新技术/前沿技术 |

### 判定标准

- **总分 ≤ 6**: **简单需求** → 走快速通道
- **总分 7-12**: **中等复杂度** → 建议完整流程，可协商
- **总分 ≥ 13**: **复杂需求** → 必须完整流程

### 判定检查清单

在开始任何工作前，评估以下问题：

```
□ 需求边界是否清晰？（模糊→复杂）
□ 是否需要设计新的API接口？（是→复杂）
□ 是否涉及数据库变更？（是→复杂）
□ 是否需要集成外部服务？（是→复杂）
□ 是否有性能/安全特殊要求？（是→复杂）
□ 是否影响现有系统的其他部分？（是→复杂）
□ 是否需要团队协作/跨模块改动？（是→复杂）
```

**判定结果应用**：
- 若用户明确要求"快速实现"、"简单处理"、"先做个原型" → 可降一级复杂度
- 若涉及核心系统、生产环境、用户数据 → 必须升一级复杂度

## 路径A：简单需求快速通道

适用于：bug修复、小功能添加、配置修改、简单重构

### 阶段A1：快速开发

**目标**：快速实现，不求完美，但求可用

**执行步骤**：

1. **理解需求**（5分钟）
   - 明确改动范围
   - 确认输入输出
   - 识别关键约束

2. **直接编码**
   - 遵循现有代码风格
   - 添加必要注释
   - 保持改动最小化

3. **即时自测**
   - 语法检查：`lsp_diagnostics`
   - 基础功能验证
   - 边界情况检查

**质量要求**：
- 代码能编译/运行
- 无明显语法错误
- 核心功能可用

### 阶段A2：基础测试

**目标**：确保基本可用性

**执行步骤**：

1. **静态检查**
   - 类型检查通过
   - 代码风格检查（如有配置）

2. **功能验证**
   - 主流程通跑
   - 关键输入验证
   - 错误处理检查

3. **回归检查**（如适用）
   - 确保未破坏原有功能

### 阶段A3：直接交付

**输出物**：
- 代码文件
- 简要说明（改动点、验证方式）
- 待办事项（如有已知问题）

**交付确认**：
```
✅ 代码已实现并测试
✅ 语法检查通过
⚠️  已知限制：[列出]
📝 建议后续优化：[列出]
```

## 路径B：复杂需求完整流程

适用于：新功能开发、系统重构、架构升级、核心模块

### 阶段B1：需求分析

**目标**：充分理解需求，识别风险，制定策略

**执行步骤**：

1. **需求澄清**
   - 识别模糊点，向用户提问
   - 明确验收标准
   - 确定优先级（Must/Should/Could/Won't）

2. **现状分析**
   - 阅读相关代码
   - 理解现有架构
   - 识别依赖和约束

3. **可行性评估**
   - 技术可行性
   - 时间可行性
   - 风险评估

4. **任务拆解**
   - 拆分为可独立执行的原子任务
   - 估算每个任务的复杂度
   - 确定任务依赖关系

**输出物**：《需求分析报告》
```markdown
# 需求分析报告

## 需求概述
- 目标：[一句话描述]
- 背景：[为什么需要]
- 范围：[包含/不包含]

## 验收标准
1. [可验证的标准1]
2. [可验证的标准2]

## 技术约束
- [约束1]
- [约束2]

## 风险评估
| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| [风险1] | 高/中/低 | 高/中/低 | [措施] |

## 任务清单
- [ ] 任务1（复杂度：X）
- [ ] 任务2（复杂度：Y）
- [ ] 任务3（复杂度：Z）

## 建议技术方案
- 方案A：[简述]
- 方案B：[简述]
- 推荐：[理由]
```

### 阶段B2：技术设计 ⬅️ 人工审查点

**⚠️ 重要：此阶段完成后必须暂停，等待用户审查确认**

**目标**：输出详细技术方案，确保方向正确

**执行步骤**：

1. **架构设计**
   - 模块划分
   - 接口定义
   - 数据流向

2. **技术选型**
   - 语言/框架选择
   - 库/工具选择
   - 理由说明

3. **详细设计**
   - API设计（路径、参数、返回值）
   - 数据模型（ER图/Schema）
   - 核心流程（时序图/流程图）
   - 错误处理策略

4. **非功能设计**
   - 性能考虑
   - 安全考虑
   - 可扩展性考虑

**输出物**：《技术方案设计文档》
```markdown
# 技术方案设计文档

## 架构设计
[架构图/模块图]

### 模块划分
1. **模块A**：职责、接口
2. **模块B**：职责、接口

## 技术选型
| 组件 | 选择 | 理由 |
|------|------|------|
| 语言 | [选择] | [理由] |
| 框架 | [选择] | [理由] |
| 数据库 | [选择] | [理由] |

## API设计

### 接口1：/api/resource
- 方法：GET/POST/PUT/DELETE
- 参数：[列表]
- 返回值：[Schema]
- 错误码：[列表]

## 数据模型
```
[ER图或表结构定义]
```

## 核心流程
```
[时序图或流程图描述]
```

## 测试策略
- 单元测试覆盖：[目标]
- 集成测试重点：[模块]
- 性能测试指标：[指标]

## 部署计划
- [ ] 步骤1
- [ ] 步骤2
```

**人工审查清单**：
```
🔍 请审查以下内容：

□ 架构设计是否合理？
□ 技术选型是否合适？
□ API设计是否满足需求？
□ 数据模型是否正确？
□ 是否有遗漏的边界情况？
□ 风险是否被充分考虑？

确认后回复："技术方案已确认，可以进入开发阶段"
或提出修改意见。
```

### 阶段B3：代码开发

**目标**：按技术方案高质量实现

**执行步骤**：

1. **开发准备**
   - 创建特性分支
   - 准备开发环境
   - 设置工具（lint、format等）

2. **按任务实现**
   - 遵循技术方案
   - 每个任务完成后自测
   - 及时提交（小步快跑）

3. **代码质量保证**
   - 类型检查：使用 `lsp_diagnostics`
   - 代码审查：自我审查 + 工具辅助
   - 文档注释：公共API必须注释

4. **持续集成**
   - 每次提交前跑本地测试
   - 保持主干可编译

**代码审查清单**：
```
□ 代码是否符合技术方案？
□ 命名是否清晰？
□ 是否有重复代码？
□ 错误处理是否完善？
□ 是否有明显性能问题？
□ 是否包含敏感信息？
□ 注释是否清晰？
```

### 阶段B4：全面测试

**目标**：确保代码质量，满足验收标准

**执行步骤**：

1. **单元测试**
   - 核心逻辑覆盖
   - 边界条件测试
   - 错误路径测试

2. **集成测试**
   - 模块间交互测试
   - API端到端测试
   - 数据流测试

3. **静态分析**
   - 代码风格检查
   - 潜在bug扫描
   - 安全漏洞检查

4. **性能测试**（如适用）
   - 基准测试
   - 压力测试
   - 内存分析

**测试报告模板**：
```markdown
# 测试报告

## 单元测试
- 测试用例数：X
- 通过率：Y%
- 覆盖率：Z%

## 集成测试
- 场景数：X
- 通过率：Y%

## 发现的问题
| 问题 | 严重程度 | 状态 | 备注 |
|------|----------|------|------|
| [问题1] | 高/中/低 | 已修复/待修复 | |

## 测试结论
[是否通过，是否可以进入交付]
```

### 阶段B5：规范交付

**目标**：完整交付，便于维护

**交付物清单**：

1. **代码**
   - 源代码
   - 配置文件
   - 测试代码

2. **文档**
   - README（快速开始、使用说明）
   - API文档（接口详情）
   - 架构文档（可选）

3. **测试**
   - 测试用例
   - 测试报告

4. **部署**
   - 部署脚本/配置
   - 回滚方案

5. **交接**
   - 改动清单
   - 待办事项
   - 已知问题

**最终确认清单**：
```
✅ 代码已实现并测试通过
✅ 文档已更新
✅ 代码审查通过
✅ 无高优先级未解决问题
✅ 可以合并/部署
```

## 人工审查检查点

### 检查点1：技术方案审查（必须）

**位置**：阶段B2完成后
**动作**：暂停执行，等待用户确认
**审查内容**：
- 架构设计合理性
- 技术选型适当性
- API设计完整性
- 风险评估充分性

**通过标准**：
```
用户回复以下之一：
- "确认" / "可以" / " proceed" 
- 提出具体修改意见（修改后再次审查）
```

### 检查点2：代码审查（可选）

**位置**：阶段B3关键里程碑后（如核心模块完成）
**触发条件**：
- 用户明确要求中间审查
- 代码量超过预期50%
- 遇到重大技术难题

### 检查点3：最终交付审查（建议）

**位置**：阶段B5完成后
**动作**：展示交付物，确认完成
**确认内容**：
- 功能是否满足需求
- 文档是否清晰
- 是否还有其他需求

## 流程切换规则

### 简单→复杂（升级）

**触发条件**：
- 开发过程中发现需求比预期复杂
- 需要引入新的技术栈
- 需要设计新的API或数据模型

**处理方式**：
1. 暂停当前工作
2. 重新进行复杂度评估
3. 切换到完整流程，从阶段B1开始
4. 保留已完成的工作作为参考

### 复杂→简单（降级）

**触发条件**：
- 用户明确要求简化
- 需求范围大幅缩小
- 技术方案比预期简单很多

**处理方式**：
1. 与用户确认简化范围
2. 跳到阶段B3（开发），跳过技术设计审查
3. 调整质量标准（按快速通道）

## 质量门禁

### 简单需求门禁

```
必须通过：
□ 代码能编译/运行
□ lsp_diagnostics无错误
□ 核心功能手动验证通过

建议通过：
□ 代码风格一致
□ 有基本注释
```

### 复杂需求门禁

```
必须通过：
□ lsp_diagnostics无错误
□ 单元测试通过率 > 80%
□ 代码审查通过（人工或工具）
□ 技术方案已确认

建议通过：
□ 单元测试覆盖率 > 70%
□ 集成测试通过
□ 文档完整
□ 性能达标
```

## 异常处理

### 需求变更

**处理流程**：
1. 评估变更影响
2. 决定是否接受变更
   - 小变更：直接调整
   - 大变更：重新走需求分析
3. 更新任务清单和时间预估
4. 告知用户影响

### 技术障碍

**处理流程**：
1. 尝试解决（30分钟）
2. 无法解决→查询资料/寻求帮助
3. 仍无法解决→向用户汇报
   - 说明问题
   - 提供替代方案
   - 请求决策

### 时间超支

**处理流程**：
1. 分析超支原因
2. 评估剩余工作量
3. 向用户汇报
   - 当前进度
   - 预计完成时间
   - 建议（加班/砍需求/延后期限）

## 最佳实践

### Do

- ✅ 始终先判定复杂度
- ✅ 简单需求不要过度设计
- ✅ 复杂需求必须经过技术设计审查
- ✅ 保持任务清单更新
- ✅ 及时与用户沟通进展
- ✅ 保留所有设计决策的理由

### Don't

- ❌ 不要在需求不清时直接编码
- ❌ 不要跳过技术设计审查（复杂需求）
- ❌ 不要在一个任务中混合多个功能
- ❌ 不要忽视静态检查的错误
- ❌ 不要交付未测试的代码
- ❌ 不要让用户猜测进度

## 示例对话

### 示例1：简单需求（快速通道）

**用户**：帮我修复一个bug，登录接口返回500错误

**AI**：
```
这是一个简单bug修复需求，走快速通道。

【阶段A1：快速开发】
□ 分析：查看登录接口代码
□ 发现：缺少空指针检查
□ 修复：添加检查逻辑

【阶段A2：基础测试】
□ 语法检查：通过
□ 功能验证：登录成功

【阶段A3：交付】
✅ 已修复登录接口500错误
✅ 添加了用户名为空的检查
⚠️  建议：补充单元测试覆盖此场景
```

### 示例2：复杂需求（完整流程）

**用户**：帮我设计一个任务调度系统，支持定时和实时触发

**AI**：
```
这是一个复杂需求，走完整流程。

【阶段B1：需求分析】
需求评估：
- 涉及模块：调度器、执行器、存储、API
- 技术难点：任务状态管理、并发控制、容错处理
- 复杂度评分：15分（复杂需求）

【阶段B2：技术设计】⬅️ 人工审查点
（输出技术方案文档）

请审查技术方案，确认后进入开发阶段。

[等待用户确认...]

【阶段B3-B5】
（按确认的方案执行开发、测试、交付）
```
